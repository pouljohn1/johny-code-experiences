## Routing fundamentals (App Router)

Next.js routes are defined by your folder structure inside `app/`.

### Static routes

- `app/about/page.tsx` â†’ `/about`
- `app/contact/page.tsx` â†’ `/contact`

### Layouts (shared UI)

- `app/layout.tsx` wraps every page
- Add nested layouts with `app/(section)/layout.tsx`

```tsx

import Link from "next/link";

      <body>

        </header>
        <div style={{ maxWidth: 720, margin: "0 auto", padding: 24 }}>{children}</div>
      </body>
    </html>
  );
}
```

### Links and navigation

Use `next/link` for client-side navigation (faster than full page reloads).

## Server Components vs Client Components

Next.js defaults to **Server Components**. They render on the server, ship less JavaScript, and can access secrets. Use **Client Components** for interactivity.

- Server Components: default, can use `async`, data fetching, no browser-only APIs
- Client Components: add at top: `"use client"`; can use state, effects, event handlers, browser APIs

### Server component data fetching

You can fetch directly in a Server Component.

```tsx
// app/page.tsx
export default async function Home() {
  const res = await fetch("https://jsonplaceholder.typicode.com/todos/1", {
    // Revalidate data at most every 60 seconds
    next: { revalidate: 60 },
  });
  const todo = await res.json();

  return (
    <main>
      <h1>Welcome</h1>
      <p>Server-fetched todo: {todo.title}</p>
    </main>
  );
}
```

Notes:

- `fetch` is enhanced by Next.js with caching and revalidation
- Omit `next.revalidate` to fully cache during build if possible

### Adding interactivity with a Client Component

Pass data from a Server Component into a Client Component for stateful UI.

```tsx
// app/components/Counter.tsx
"use client";
import { useState } from "react";

export default function Counter({ start = 0 }: { start?: number }) {
  const [count, setCount] = useState(start);
  return (
    <button onClick={() => setCount((c) => c + 1)}>Count: {count}</button>
  );
}
```

```tsx
// app/page.tsx (using Counter)
import Counter from "./components/Counter";

export default async function Home() {
  // Could fetch data here if needed
  return (
    <main>
      <h1>Interactive Example</h1>
      <Counter start={1} />
    </main>
  );
}
```

### Environment variables

- Put secrets in `.env.local` (not committed)
- Server-only vars (default): `DATABASE_URL=â€¦`
- Expose to client with `NEXT_PUBLIC_`: `NEXT_PUBLIC_API_BASE=â€¦`

Access on the server via `process.env.DATABASE_URL`, and on the client only for `NEXT_PUBLIC_*`.

## Creating an API route (Route Handlers)

Use route handlers in the App Router.

```ts

  return NextResponse.json({ message: "Hello from Next.js API" });

```

- Available methods: `GET`, `POST`, `PUT`, etc.
- Access at `/api/hello`

## Deployment

### Deploy to Vercel (recommended)

1) Push your code to GitHub/GitLab/Bitbucket.

2) Create a Vercel account and "Add New Project" â†’ import your repo.

3) Vercel auto-detects Next.js. Keep defaults; set any environment variables.

4) Click Deploy. Youâ€™ll get a preview URL and a production URL after merging to main.

- Edge/network features and Server Components work out of the box
- Automatic SSL, CDN, and image optimization

### Self-hosting with Node.js

1) Build your app:

```bash
npm run build
```

2) Start the production server:

```bash
npm run start
# serves on http://localhost:3000
```

3) Set environment variables in your host (e.g., `.env` or platform config).

Note: For Docker or advanced setups, consider `output: "standalone"` in `next.config.js` to simplify deployment artifacts.

## Common pitfalls and tips

- Add `"use client"` only at the very top of a file; not inside blocks
- Client Components can use state/effects; Server Components cannot
- Do not use `window` or browser APIs in Server Components
- Use `next/link` for navigation (avoid plain `<a>` for internal routes)
- Dynamic route params come from the folder name: `[id]`, `[slug]`, etc.
- Use `.env.local` for secrets; only expose with `NEXT_PUBLIC_` if needed on the client
- Prefer data fetching in Server Components for better performance
